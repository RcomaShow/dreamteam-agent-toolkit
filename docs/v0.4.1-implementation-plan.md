# DreamTeam 0.4.1 Implementation Plan

Status: implemented as one reviewable commit on top of DreamTeam 0.4.0.

## Goal

Make the existing architecture usable without weakening its invariants. A new user should be able to install the Claude Code plugin, generate a valid advisory configuration, diagnose readiness, understand whether durable status is available, and run a first routed task without first learning the internal ledger or handoff protocols.

## Architectural decisions

1. **No additional Sonnet layer.** `execution-sonnet-lead` already acts as the bounded Sonnet implementer for delegated L1/L2 work. `verification-independent-reviewer` remains a distinct agent identity and acceptance oracle. The public documentation now describes these roles explicitly.
2. **Strict enforcement remains opt-in.** The default configuration is telemetry disabled, ledger off, and advisory enforcement. Strict mode enables SQLite telemetry and still requires working Claude Code hooks.
3. **No in-memory operational ledger.** Claude Code hooks run as separate processes; an in-memory ledger would not preserve reservations, charges, checkpoints, or contract bindings between hook events. Durable mode therefore remains SQLite.
4. **Zero runtime dependencies.** Standard Python packaging is added, while the runtime continues to use only the standard library.
5. **Wire compatibility is preserved.** Config version 2, route names, DCP/2, CHP/2, agent identifiers, and benchmark bucket fields are unchanged.

## Implemented scope

### Onboarding

- Add `dreamteam init` and `/dreamteam:init`.
- Generate `dreamteam.config.json` atomically.
- Default to `lean`, `balanced`, advisory enforcement, and no telemetry.
- Refuse silent overwrite and reject symlink targets.
- Add `dreamteam.config.minimal.json` as a validator-checked golden configuration.

### Diagnostics

- Add deterministic `dreamteam doctor` output in text or JSON.
- Validate configuration, plugin version, hook packaging, agent catalog, strict readiness, and ledger location.
- Emit stable issue codes and copyable remediation instructions.
- Explain that Opus-Sonnet uses a bounded Sonnet implementer and a different Sonnet reviewer identity.

### Status

- Add `dreamteam status` and `/dreamteam:status`.
- Read SQLite in read-only mode.
- Report only metadata: run ID, config hash, charges, reservations, active workers, checkpoint counts, failed tool events, and invalidation categories.
- Never expose source content, raw commands, prompts, or invalidation details.
- Explain clearly when telemetry is disabled, the ledger is absent, or an explicit run ID is required.

### Packaging and developer experience

- Add `pyproject.toml`, a typed-package marker, and a `dreamteam` console entry point.
- Add a Makefile for canonical check and release commands.
- Keep development tools optional and retain zero runtime dependencies.

### Documentation and release

- Restructure the README around a five-minute first run.
- Add init and status commands to the Claude Code adapter documentation.
- Bump runtime, plugin, smoke tests, validation, and release documentation to 0.4.1.
- Extend artifact smoke testing to cover init, doctor, and status.

## Explicitly out of scope

- New topologies or model families.
- Renaming existing machine identifiers.
- Provider executors or paid inference.
- Changes to routing economics or benchmark claim gates.
- A durable controller, recovery state machine, or additional adapters.
- Universal cost-saving claims.

## Acceptance gates

The commit is acceptable only when all of the following pass:

```bash
python scripts/sync_claude_adapter.py
git diff --exit-code
python scripts/validate.py
python -m unittest discover -s tests -v
python scripts/measure.py
python -m compileall dreamteam adapters/claude-code/plugins/dreamteam
python scripts/build_release.py
python scripts/smoke_plugin_artifact.py dist/dreamteam-claude-code-plugin-0.4.1.zip
```

Additional invariants:

- workers cannot spawn workers;
- writer and acceptance oracle remain different identities;
- C3 and public-contract ownership remains executive-owned;
- Opus-Sonnet contains no hidden Haiku stage;
- strict mode cannot report ready without confirmed hooks and SQLite telemetry;
- generated adapter files remain byte-identical to canonical runtime sources;
- no empirical savings claim is introduced.
