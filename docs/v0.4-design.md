# DreamTeam 0.4 — Fail-Closed Orchestration

Version 0.4 converts several 0.3 policies into runtime invariants without adding a distributed framework.

## Execution model

The root session remains the only physical dispatcher. The router returns a concrete agent role and ordered execution chain. Supported topologies are:

- Lean: Sonnet executive → bounded Haiku or Sonnet role;
- Opus-Sonnet: Opus executive → Sonnet implementation lead or reviewer;
- Frontier: Opus executive → Sonnet lead/reviewer → Haiku worker.

Every active stage must have a non-zero forecast. Hidden stages invalidate the candidate. C3 remains executive-owned.

## Safety model

Strict mode binds a run to one project-root config hash, requires hooks plus SQLite, confines paths to non-symlink files below the root, denies protected-config writes, and keys dispatch reservations by tool-use ID. Bash is denied unless the command is a bundled deterministic DreamTeam wrapper or is structurally safe and its exact SHA-256 is explicitly permitted. Reads are staged at PreToolUse and committed only when the blob remains stable. DreamTeam CHP/2 output is validated at SubagentStop before acceptance.

## Accounting

Reservations protect future budget. Charges represent incurred cost and are never released. Reconciliation atomically replaces a reservation with a durable charge. Direct, delegated, and C3 routes all obey the hard run cap.

## Protocol model

DCP/2 and CHP/2 are parsed by the installable runtime. A valid DCP is registered immutably in the SQLite ledger before dispatch; strict handoffs must match that run/task contract binding, current source anchors, and any registered writer/reviewer identities.

## Evidence and claims

Benchmark API-equivalent cost is recomputed from the immutable pricing snapshot and per-model usage. Pair invariants cover task, commit, oracle, environment, config, timeout, cache cohort, and adapter version. Publication is fail-closed per task bucket.
