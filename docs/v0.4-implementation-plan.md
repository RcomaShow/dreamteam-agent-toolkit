# DreamTeam 0.4 Implementation Plan

This plan maps the architecture and security review into an incremental release. Each phase leaves the repository executable and independently testable.

## Phase 1 — P0 input, config, filesystem, shell, and release hardening

**Goals**

- reject coercive JSON booleans, unknown fields, and non-finite numbers;
- require coherent strict telemetry and hooks;
- resolve every path against one stable project root;
- protect runtime configuration and deny unaccountable shell reads;
- build archives only from reviewed Git files.

**Acceptance**

- outside-root, traversal, symlink, config-write, composite-shell, nested-agent, and unidentifiable dispatch fixtures fail closed;
- `.env`, untracked files, dirty changes, and symlinks cannot enter a release archive.

## Phase 2 — P1 executable routing and durable accounting

**Goals**

- make profiles and model aliases executable;
- add `opus-sonnet` alongside Lean and Frontier;
- return concrete roles and ordered chains;
- apply budget caps to direct, delegated, and C3 paths;
- separate reservations from incurred charges.

**Acceptance**

- every topology accounts for each active model exactly once;
- Opus-Sonnet rejects hidden Haiku usage;
- writes require a distinct non-zero-cost reviewer;
- charged cost remains consumed after reservation release.

## Phase 3 — P1 protocol and benchmark integrity

**Goals**

- ship the DCP/2–CHP/2 parser in the plugin and register DCP hashes immutably before dispatch;
- bind run, task, contract, anchors, author, and reviewer;
- recompute benchmark costs and enforce pair equivalence;
- apply publication gates per bucket.

**Acceptance**

- malformed, stale, mismatched, self-reviewed, unregistered, or unbound handoffs are rejected;
- fabricated API-equivalent cost and mismatched benchmark arms cannot produce claims.

## Phase 4 — P2 quality, CI, documentation, and observability

**Goals**

- expand adversarial and lifecycle tests;
- validate generated drift and the exact release archive;
- document metrics for token, cost, latency, context, reliability, and security;
- keep provider execution opt-in and non-bundled.

**Acceptance**

- full unit, structural, compile, build, and isolated smoke suites pass;
- the release manifest matches the source commit and archive contents.

## Deferred P3 work

The following improvements remain intentionally outside 0.4 to avoid overengineering: a durable DAG controller, semantic repository cache, worktree-isolated writers, confidence-bound online calibration, MCP tool gateway, signed release provenance, and additional provider adapters. The 0.4 interfaces and event identifiers are designed so these can be added without changing constitutional ownership or protocol semantics.

## Rollback

Each security gate supports advisory mode for diagnosis, but strict mode never silently downgrades. The public 0.3 `choose_route` compatibility alias remains available. Configuration version stays at 2; the new topology and output fields are additive. Benchmark input is intentionally a new strict schema and must be migrated explicitly.
